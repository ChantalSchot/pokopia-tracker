2. Assumptions
1.	Flavour favourites are the 6 "flavour" names found in pokemon.json but absent from favourites.json: sweet flavors, sour flavors, spicy flavors, bitter flavors, dry flavors, and none. These are used as the 6th favourite for some Pokémon and have no items linked to them. They are treated as a separate FlavourFavourite type (or a flag isFlavour = true) and are never addable to houses.
2.	Pokémon with favourites: ["none"] (e.g. Ditto) is treated as having 1 flavour-only favourite. Since the prompt says "exactly 6 favourites", this appears to be a data quality issue — see contradictions section below.
3.	Items are linked to favourites by name — favourites.json contains arrays of item names under each favourite. The mapping relationship is favourite ↔ items[] by name string. This is the canonical source for which item activates which favourite. Items not appearing in any favourite list (most items) are none type for favourite purposes.
4.	31 item names in favourites.json do not match any name in items.json (e.g. Pokémetal, Poké Ball light — likely encoding artifacts or items not yet in the dataset). These will be imported as-is and logged as warnings during import; no match in items.json means they have no item entity counterpart.
5.	Favourite-to-item linking will be stored in the database as a join table favourite_items derived from favourites.json. The favourites.json items array = item names that activate this favourite.
6.	timeOfDay field in pokemon.json has garbled values (e.g. " ", "A", "a", "d", "l", "y"). These appear to be data corruption. Valid values are All day, Daytime, Evening, Morning, Nighttime. Corrupted values will be stored as-is and flagged in import logs; a note will be in docs.
7.	rarity value "None" will be included as an enum value NONE.
8.	litterDrop value "" (empty string) = no litter drop; stored as null or enum NONE.
9.	Regions are derived from pokemon.json regions array, not from a separate dataset. The valid regions are: Bleak Beach, Cloud Island, Palette Town, Rocky Ridges, Sparkling Skylands, Withered Wastelands. They will be represented as a Region enum.
10.	House capacity for kit type is determined by the housing-kit's size field (integer 1, 2, or 4).
11.	idealHabitat field on a house is called "habitat" in the UI but maps to the same IdealHabitat enum used on Pokémon (Bright, Cool, Dark, Dry, Humid, Warm). This is separate from the imported Habitat master data entities.
12.	Password reset via email will use Spring Mail + a simple token table (no 3rd-party service assumed). The user must configure SMTP in .env.
13.	Asset files exist in resources/assets/ subfolders mirroring the JSON imagePath/spritePath values. The Spring Boot backend will serve static assets from a configured directory, and the frontend will reference them via the backend's static asset base URL.
14.	Admin seeding: the first admin is set manually in the database (documented); no admin self-registration endpoint.
15.	JWT-based stateless auth is used, with access tokens (short-lived, ~15 min) and refresh tokens (long-lived, ~7 days) stored in HttpOnly cookies to mitigate XSS.
 
3. Contradictions / Ambiguities / Questions
A — Flavour favourites and the "exactly 6" rule
The prompt says each Pokémon has exactly 6 favourites and one is a flavour favourite. However:
•	Several Pokémon in pokemon.json have only 6 entries, some of which are clearly flavour names (e.g. sweet flavors) — so the rule is satisfied.
•	Ditto has favourites: ["none"] — only 1 favourite. This contradicts "exactly 6".
•	Some other Pokémon may have similar issues.
Recommended decision: Treat "none" as the flavour favourite. For Pokémon with fewer than 6 favourites, treat missing entries as data quality issues and log them during import without failing. Document this explicitly.
B — Favourite-to-item relationship direction
favourites.json lists item names under each favourite. items.json has no favourite field. There are 31 item name references in favourites.json that don't exist in items.json (likely Pokémon encoding issues in item names, e.g. Pokémetal vs actual item name).
Recommended decision: Build the join table from favourites.json. Log mismatches as warnings. No import failure. Admin can reload individual datasets to fix.
C — Item type "Road" vs UI grouping
The prompt says items in houses are split by decoration, relaxation, toy, or none. items.json also has type: "Road" and type: "Food". The UI rule mentions only 4 groupings.
Recommended decision: Map Road and Food to the none/untyped category in house item display. The stored type field retains all original values; the grouping is a display concern.
D — timeOfDay data corruption
pokemon.json has garbled timeOfDay values (single letters, spaces). These cannot be mapped to valid enum values.
Recommended decision: Store raw string value in the database (not as an enum, or as a nullable enum with an UNKNOWN fallback). Log corrupt values during import. This can be cleaned up by an admin after import.
E — Multiple Pokémon with the same number
The prompt notes habitats link Pokémon by number string, and multiple Pokémon can share the same number. Data analysis shows 310 Pokémon — it should be verified if duplicates exist.
Recommended decision: number is not unique in the database schema. Habitat-to-Pokémon linkage is done by matching number string to pokemon.number, returning all matches.
F — Password reset email infrastructure
The prompt requires password reset via email but does not specify a mail provider.
Recommended decision: Use Spring Mail with configurable SMTP (documented in .env.example). If SMTP is not configured, the reset token is logged to server logs in development mode.
G — idealHabitat as enum — housing-kits.json has no idealHabitat
Housing kits don't have an idealHabitat; only Pokémon and (optionally) houses do. The house's idealHabitat is independently set by the user from the enum.
H — House size for custom type
Custom houses have capacity 4. size is never directly editable. Width/depth/height are editable for custom houses, but size is derived or fixed at null/not applicable.
Recommended decision: size is stored only for kit and habitat type houses (copied from their source at creation). For custom houses, size is null and capacity is hardcoded to 4.
I — habitat type house and habitat dataset coupling
The prompt says a habitat type house "references a habitat record for source information such as size" but capacity is 1 for habitat houses — habitat records don't have a size field in habitats.json.
Recommended decision: habitat type houses have a reference to a Habitat entity (optional FK), used for display/informational purposes only. Capacity is always 1 for habitat houses regardless of the habitat record.

7. Proposed Auth/Authz Design
•	Mechanism: JWT stateless with short-lived access token (15 min) + long-lived refresh token (7 days), both in HttpOnly, SameSite=Strict cookies (mitigates XSS and CSRF)
•	Registration: POST /api/auth/register — open endpoint
•	Login: POST /api/auth/login — returns tokens in cookies
•	Logout: POST /api/auth/logout — clears cookies + invalidates refresh token in DB
•	Token refresh: POST /api/auth/refresh — validates refresh token, issues new access token
•	Password reset: POST /api/auth/forgot-password + POST /api/auth/reset-password?token=...
•	Role checks: @PreAuthorize("hasRole('ADMIN')") on admin endpoints; route guards on Angular side
•	Password rules: minimum 8 characters, at least 1 uppercase, 1 digit; BCrypt hashing (strength 12)
•	Future extensibility: Token table is in place for refresh; User.emailVerified boolean prepared; 2FA boolean flag reserved

11. Edge Cases and Business Rule Clarifications
1.	Homeless Pokémon: Registered + no house assignment. Shown on dashboard. Pokédex filter available.
2.	Reassigning a housed Pokémon: Moving to a new house automatically removes it from the previous house. UI must make this explicit with a confirmation prompt.
3.	House region change: If user changes a house's region, a modal asks: "Move Pokémon with the house to [new region]?" Those that don't move become homeless.
4.	Deleting a house: Assigned Pokémon become homeless. Confirmation dialog required.
5.	House name uniqueness: Enforced per user + region combination. Same name allowed in different regions.
6.	Suggestions logic:
•	House is full → no suggestions
•	House has items → suggest Pokémon whose idealHabitat matches AND whose non-flavour favourites overlap with the active favourites (from items) — ranked by overlap count descending
•	House has no items → suggest Pokémon whose idealHabitat matches AND whose non-flavour favourites overlap with the non-flavour favourites of Pokémon already in the house — ranked by overlap count
•	Tie-breaking: alphabetical by Pokémon name
•	Exclude already-registered-and-housed Pokémon
7.	Warning state for Pokémon in house: If none of the Pokémon's non-flavour favourites are active in the house (i.e. no item in the house links to any of that Pokémon's favourites), show a warning. Flavour favourites are excluded from this check — they never activate and never count for or against the warning.
8.	Flavour favourite in house: Never addable to a house. UI must not show flavour favourites in the "active favourites" list or suggest items for them.
9.	Capacity enforcement: Assignment rejected with 409 Conflict if house is at or over capacity.
10.	Duplicate registration: Registering an already-registered Pokémon is a no-op or returns 200 OK (idempotent).
11.	Admin data reload: Reload is upsert; user-owned data (houses, registrations) is never affected by import reload.
12.	timeOfDay corrupted values: Imported as raw string UNKNOWN or stored verbatim; shown in UI as "—" or "Unknown".
13.	Ditto's single "none" favourite: Imported as-is with isFlavour=true. No warning during import as this is the known data state.
14.	Item-favourite mismatches (31 names): Logged at WARN. Items exist in the DB but have no favourite link for those specific names. No functional impact.

Key implementation details:
•	JWT: io.jsonwebtoken (jjwt 0.12.x). Access token 15min, refresh token 7 days. Both in HttpOnly cookies named access_token and refresh_token. Inactivity: update lastUsedAt on refresh token, consider 3hr inactivity as expired.
•	Spring Security: permit /api/auth/, /swagger-ui/, /v3/api-docs/**. All others require auth. Admin endpoints require ADMIN role.
•	Flyway migrations in src/main/resources/db/migration/: V1__init_schema.sql (all tables), V2__constraints.sql (if needed)
•	Import: DataImporter runs at startup (ApplicationRunner) if pokemon table empty. Upsert logic: saveIfAbsent or deleteAllAndSave per dataset. Unicode normalization for item-favourite matching.
•	Ditto: After importing pokemon, find the one with name="Ditto", set isDitto=true, clear its favourites, save.
•	TimeOfDay: filter imported values, only keep: "All day", "Daytime", "Evening", "Morning", "Nighttime"
•	Suggestion endpoint: GET /api/houses/{id}/suggestions returns list of PokemonResponse
•	Active favourites: GET /api/houses/{id}/active-favourites returns list of FavouriteResponse
•	House capacity: computed method in House entity. KIT → housingKit.size, HABITAT/CUSTOM → 1
•	House region change payload: ChangeHouseRegionRequest { newRegion: Region, pokemonIdsToMove: List<UUID> }
•	Warning: Pokemon in house has no non-flavour favourites matching any active favourite → warn=true field on UserPokemonResponse
•	Error response: { timestamp, status, error, message, path }
•	Pagination: all list endpoints return PageResponse<T> with { content, totalElements, totalPages, page, size }
•	OpenAPI: springdoc-openapi-starter-webmvc-ui:2.x, available at /swagger-ui.html
•	CORS: Allow http://localhost:4300 in dev

Routes (app.routes.ts):
•	/ → redirect to /login if not authed, /dashboard if authed
•	/login, /register, /forgot-password, /reset-password → public
•	/dashboard, /pokedex, /houses, /items, /habitats, /profile → authGuard
•	/admin/** → authGuard + adminGuard
Auth flow:
•	AuthService stores user state (role, username) in memory (from JWT claims decoded)
•	HTTP interceptor automatically includes credentials: true (for cookies)
•	401 response → token refresh attempt → retry → if still 401 → logout
•	AuthGuard: checks AuthService.isLoggedIn(); if not → navigate to /login
•	AdminGuard: checks AuthService.isAdmin(); if not → navigate to /dashboard
